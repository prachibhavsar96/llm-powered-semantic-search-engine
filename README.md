# LLM-Powered Q&A Search Engine

A full-stack semantic search application that allows users to upload documents and perform intelligent search using Pinecone for vector indexing, Sentence Transformers for generating embeddings, and Google Gemini (via LangChain) for query understanding and answer generation.

Built with FastAPI, PostgreSQL, React, Sentence Transformers, Pinecone, and Gemini.

## Features

- JWT Authentication (Login / Signup)
- Google OAuth Login
- Semantic document search using embeddings
- PDF, DOCX, and TXT upload support
- User-specific private document storage
- Real-time vector search via Pinecone
- LLM-powered query understanding and answer generation (Gemini + LangChain)
- Graceful fallback to rule-based summarization if the LLM call fails
- In-memory caching for repeated searches
- Interactive Swagger API documentation

---

## Performance

Load-tested with Locust: 50 concurrent users sustained over a 2-minute run against the `/search` endpoint.

- 1,210 requests, 0 failures (100% success rate)
- 19ms median response time, 69ms p95, 22,000ms p99
- ~16 requests/second sustained throughput

Note: a small number of outlier requests (~1%) saw higher latency (up to ~24s), consistent with occasional Gemini API response time variance вЂ” worth monitoring in a production deployment.

## Setup Instructions

### 1. Clone Repository

```bash
git clone https://github.com/prachibhavsar96/llm-powered-semantic-search-engine.git
cd llm-powered-semantic-search-engine
```

---

### 2. Create Virtual Environment

```bash
python -m venv venv
```

Activate environment:

#### Windows

```bash
.\venv\Scripts\Activate.ps1
```

#### macOS/Linux

```bash
source venv/bin/activate
```

---

### 3. Install Backend Dependencies

```bash
pip install -r requirements.txt
```

---

### 4. Configure PostgreSQL

Create database:

```sql
CREATE DATABASE llm_search_engine;
```

Create `.env` in project root:

```env
DATABASE_URL=postgresql://postgres:your_password@127.0.0.1:5432/llm_search_engine
SECRET_KEY=your_secret_key
GOOGLE_CLIENT_ID=your_google_client_id
PINECONE_API_KEY=your_pinecone_api_key      # From the Pinecone dashboard (app.pinecone.io)
GOOGLE_API_KEY=your_google_api_key          # From Google AI Studio (aistudio.google.com)
```

---

### 5. Run Backend

```bash
python -m uvicorn app.main:app --reload
```

Backend runs at:

```text
http://127.0.0.1:8000
```

Swagger Docs:

```text
http://127.0.0.1:8000/docs
```

---

### 6. Setup Frontend

```bash
cd frontend
npm install
```

Create `frontend/.env`

```env
VITE_GOOGLE_CLIENT_ID=your_google_client_id
```

Run frontend:

```bash
npm run dev
```

Frontend runs at:

```text
http://localhost:5173
```

## Search Workflow

1. Upload document
2. Extract text
3. Generate embeddings
4. Store chunks in PostgreSQL and upsert vectors into Pinecone
5. Rewrite the query and perform vector search against Pinecone
6. Generate an answer with Gemini, or fall back to rule-based summarization
7. Return ranked semantic matches

## Notes

- Embeddings are stored as JSON text in PostgreSQL as a fallback, and as vectors in Pinecone for search.
- Documents and searches are scoped to the authenticated user.
- Search retrieves candidates from Pinecone, combines the similarity score with keyword overlap, filters duplicate chunks, and returns the strongest useful matches.
- The top search result includes an `answer_summary` generated by Gemini from the best matching chunks, falling back to a rule-based 2-3 sentence summary if the LLM call fails.
- Search responses include execution time, scanned document count, and cache status.
- The in-memory search cache resets when the server restarts.
- The first embedding request may take longer because the local model may need to load.

## Known Limitations

- Free-tier Pinecone index pauses after 3 weeks of inactivity and may need to be resumed from the Pinecone dashboard
- LLM answer generation depends on Gemini API availability; falls back to extractive summarization on failure
